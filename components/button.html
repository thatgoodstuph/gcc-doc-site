Button

Status: Stable
Last reviewed: 2025-01-15
Owner: Design Systems
Source links:
  • Figma
  • Implementation: Not available yet

The Button component is the primary interactive element for triggering actions in GCC products. Use it for form submissions, navigation, and calls-to-action across checkout, account management, product detail pages, and all user workflows.

Overview

The Button component is the primary interactive element for triggering actions in GCC products. It provides clear visual affordances and communicates the action that will occur when clicked or activated. Buttons are used throughout the GCC design system for form submissions, navigation, calls-to-action, and user-initiated workflows.

The Button component supports two visual styles (Primary and Secondary), multiple color variants, two size options (36px and 48px tall), and optional leading or trailing icons. All buttons include comprehensive state management for default, hover, focus, and disabled states, ensuring consistent interaction patterns across all GCC products including Amway+, Wellbeing+, account management, and checkout flows.

Usage

When to use

  • For primary actions on a page (e.g., "Add to Cart", "Submit Order", "Save Changes").
  • When the action requires user confirmation or triggers a state change.
  • For form submissions and data entry workflows.
  • For calls-to-action in marketing and product pages.
  • For navigation within workflows (e.g., "Next Step", "Continue to Checkout").
  • For destructive actions that require explicit user intent (e.g., "Delete Account", "Remove Item").

When not to use

  • For navigation between pages (use a link instead).
  • For tertiary or low-priority actions (use a text link or ghost button variant).
  • For icon-only actions without labels (use Icon Button component instead).
  • For actions that don't require user confirmation (consider inline interactions).

Dos & Don'ts

Do

  • Use clear, action-oriented labels ("Save Changes", not "OK").
  • Limit to one primary button per screen section to maintain clear hierarchy.
  • Use appropriate button colors that match the action's semantic meaning (e.g., Dark Red for destructive actions).
  • Provide sufficient color contrast between text and background for accessibility.
  • Use icons to enhance clarity when appropriate (e.g., shopping cart icon for "Add to Cart").
  • Maintain consistent button sizing within a single interface or workflow.

Don't

  • Don't use vague labels like "Click Here" or "Submit" without context.
  • Don't stack multiple primary buttons vertically without clear hierarchy.
  • Don't use buttons for navigation—use links for page-to-page navigation.
  • Don't mix button sizes arbitrarily—use 48px for primary CTAs, 36px for secondary actions.
  • Don't override button colors without design approval—stick to the defined color palette.
  • Don't disable buttons without providing clear feedback about why they're disabled.

Content guidance

  • Labels: Use sentence case for button labels. Keep labels concise (1-3 words when possible).
  • Icons: Icons are optional and should enhance, not replace, the button label. Use Font Awesome 6 Pro icons.
  • Truncation: Buttons should expand to fit their content. Avoid truncating button labels—use shorter labels instead.
  • Tone: Button labels should be action-oriented and direct. Use verbs when possible (e.g., "Add", "Save", "Delete").

Anatomy

Each Button component consists of the following parts:

  • Container (required) – The clickable surface with padding, background, border, and border-radius. Height is either 36px or 48px.
  • Leading Icon (optional) – Icon displayed before the label text. Uses Font Awesome 6 Pro, 16px size.
  • Label (required) – Text describing the action. Uses GT Walsheim Pro Bold, 16px size, 24px line height.
  • Trailing Icon (optional) – Icon displayed after the label text. Uses Font Awesome 6 Pro, 16px size.
  • Focus Ring (conditional) – Visible outline when button receives keyboard focus. 2px border, rounded to match button shape.

Spacing: Horizontal padding is 16px on both sides. Gap between icon and text is 8px when icons are present.

Behavior & States

Buttons support the following states and interactions:

States

  • Default: Neutral background and text color based on button variant. Fully interactive and ready for user interaction.
  • Hover: Background color darkens (for Primary buttons) or inverts (for Secondary buttons). Cursor changes to pointer. Text color may change to maintain contrast.
  • Focus: Visible focus ring appears (2px border, color matches button border or primary color). Button remains fully interactive. Focus ring is essential for keyboard navigation accessibility.
  • Pressed/Active: Background color darkens further from hover state. Button remains interactive until click is released.
  • Disabled: Reduced opacity, no pointer events, grayed-out appearance. Background uses Disabled Gray (#f4f4f4 for Primary, white for Secondary). Text uses Disabled Gray (#d9d9d9 for Primary, #707070 for Secondary). Button is not interactive.

Keyboard behavior

  • Buttons are focusable via Tab key navigation.
  • Buttons activate on Enter or Space key press.
  • Focus ring must be clearly visible for keyboard users (WCAG 2.1 Level AA requirement).
  • Focus order should follow logical workflow sequence.

Responsive behavior

  • On mobile devices, buttons may expand to full width in forms for easier tapping.
  • Button height remains consistent (36px or 48px) across all breakpoints.
  • Text size and icon size remain constant—do not scale down on smaller screens.

Variants

The Button component supports multiple visual and size variants:

Type variants

  • Primary: Filled background with contrasting text. High emphasis. Use for the most important action on a page (e.g., "Add to Cart", "Submit Order", "Save Changes").
  • Secondary: Outlined border with transparent or white background. Medium emphasis. Use for secondary actions or when you need visual hierarchy (e.g., "Cancel", "View Details", "Save for Later").

Size variants

  • 36px Tall: Compact size for secondary actions, toolbars, and dense interfaces. Use when space is limited or for less critical actions.
  • 48px Tall: Standard size for primary CTAs and most use cases. Provides optimal touch target size for mobile devices. Use for primary actions and hero CTAs.

Color variants

Buttons support multiple color options for different semantic meanings:

  • Black: Default primary color for most actions. Use for standard primary buttons.
  • Primary Purple: Brand color for primary actions. Use for key brand moments and high-priority CTAs.
  • Purple / Light Purple: Alternative purple variant for primary buttons.
  • Dark Blue: Use for informational or data-related actions.
  • Dark Green: Use for positive or success-related actions.
  • Dark Orange: Use for warning or attention-requiring actions.
  • Dark Red: Use for destructive or dangerous actions (e.g., "Delete", "Remove").
  • Dark Yellow: Use for cautionary or alert-related actions.
  • White: Use for secondary buttons on dark backgrounds or when you need a subtle appearance.
  • White + Purple: White background with purple text. Use for secondary actions with brand emphasis.
  • White + Black: White background with black text. Use for secondary actions on colored backgrounds.
  • Disabled Gray: Automatically applied when button is disabled. Not user-selectable.

Icon variants

  • Text only: Button with label only, no icons.
  • Leading icon: Icon appears before the label text (e.g., shopping cart icon for "Add to Cart").
  • Trailing icon: Icon appears after the label text (e.g., arrow icon for "Continue").
  • Both icons: Icons on both sides (rare, use sparingly).

Accessibility

Buttons must meet the following accessibility requirements:

Semantic element

  • Use the native <button> element, not <div> or <span>.
  • Buttons that trigger navigation should use <a> styled as buttons, but this is not recommended—prefer actual links for navigation.

ARIA attributes

  • If the button triggers a disclosure (e.g., dropdown, modal), use aria-expanded="true/false".
  • If the button has no visible label (icon-only), provide aria-label with a descriptive text.
  • For loading states, use aria-busy="true" and aria-label to indicate the action in progress.
  • For disabled buttons, the disabled attribute is sufficient—no additional ARIA needed.

Keyboard requirements

  • Buttons must be focusable via Tab key navigation.
  • Buttons must activate on Enter or Space key press.
  • Focus ring must be clearly visible (2px border, sufficient color contrast).
  • Focus order should follow logical workflow sequence.

Contrast requirements

  • Text and background must meet WCAG AA contrast ratio (4.5:1 for normal text, 3:1 for large text at 18px+).
  • Focus ring must be clearly visible against all backgrounds (minimum 3:1 contrast).
  • Disabled state must still be readable (minimum 3:1 contrast for disabled text).

Component-specific a11y rules

  • Button labels must be descriptive and action-oriented—avoid generic labels like "Click Here".
  • Icon-only buttons must have aria-label describing the action.
  • Buttons in forms should be associated with the form using proper form attributes.
  • Loading states should not remove the button from the accessibility tree—use aria-busy instead.

Typography tokens

  • typography.fontFamily.text – "GT Walsheim Pro"
  • typography.fontWeight.bold – 700 (Bold)
  • typography.fontSize.text-md – 16px
  • typography.lineHeight.text-md – 24px

Spacing tokens

  • spacing.md – 16px horizontal padding
  • spacing.sm – 8px gap between icon and text
  • spacing.button-height-sm – 36px (small button height)
  • spacing.button-height-md – 48px (standard button height)

Radius tokens

  • radius.full – 100px (fully rounded buttons)

Elevation tokens

  • Buttons do not use elevation/shadow tokens—they are flat design elements.

Theming

  • Customize button colors by overriding color tokens in your theme.
  • Do not hard-code hex values like #0066CC directly in component styles.
  • All button colors should reference design tokens from the Color foundation.

See Color for full color token reference, Typography for typography tokens, and Spacing for spacing tokens.

Code & API

Button components can be implemented using the GCC component library. Example usage:

Basic usage

import { Button } from "@gcc/components";

// Primary button (default)
<Button variant="primary" onClick={handleSubmit}>
  Add to Cart
</Button>

// Secondary button
<Button variant="secondary" onClick={handleCancel}>
  Cancel
</Button>

Advanced usage with icons and states

import { Button } from "@gcc/components";
import { ShoppingCartIcon } from "@gcc/icons";

// Primary button with leading icon
<Button 
  variant="primary"
  size="48px"
  color="black"
  leadingIcon={<ShoppingCartIcon />}
  onClick={handleAddToCart}
>
  Add to Cart
</Button>

// Secondary button with trailing icon and loading state
<Button 
  variant="secondary"
  size="36px"
  color="primary-purple"
  trailingIcon={<ArrowRightIcon />}
  loading={isSubmitting}
  disabled={!isFormValid}
  onClick={handleSubmit}
>
  Continue
</Button>

// Destructive action button
<Button 
  variant="primary"
  size="48px"
  color="dark-red"
  onClick={handleDelete}
>
  Delete Account
</Button>

Examples

Common button usage patterns in GCC products:

E-commerce workflows

  • Product Detail Page: Primary Black button, 48px, "Add to Cart" with shopping cart icon.
  • Cart Page: Primary Black button, 48px, "Proceed to Checkout" with arrow icon.
  • Checkout: Primary Purple button, 48px, "Place Order" as the final CTA.

Account management

  • Profile Settings: Primary Black button, 48px, "Save Changes" for form submission.
  • Account Deletion: Primary Dark Red button, 48px, "Delete Account" for destructive action.
  • Secondary Actions: Secondary Black button, 36px, "Cancel" or "Back" for navigation.

Form submissions

  • Primary Action: Primary button, 48px, with clear action label (e.g., "Submit", "Save", "Continue").
  • Secondary Action: Secondary button, 36px, for cancel or alternative actions.
  • Loading State: Button shows loading spinner and disables during async operations.

Refer to the Figma Button Component file for visual examples of all button variants, states, and color combinations.
