Grid System

The GCC grid system defines responsive layout scaffolding across mobile, tablet, and desktop viewports using consistent columns, gutters, and margins.

Overview

The GCC grid system provides a consistent layout framework for pages and components across all products. It defines how content aligns to columns, gutters, and margins at three primary breakpoints: Mobile (375px), Tablet (768px), and Desktop (1280px), as documented in the Figma Grid System.

Use the grid system for all page-level and major component layouts so that experiences feel aligned, readable, and predictable across breakpoints. The grid ensures consistent spacing and alignment whether content is viewed on mobile devices, tablets, or desktop screens.

Usage

When to use

  • Page layouts (home, PDP, PLP, account, checkout, Wellbeing+ flows).
  • Content sections composed of multiple components (e.g., hero + cards + tables).
  • Any layout that must adapt responsively from mobile to desktop.
  • Multi-column content areas that need consistent alignment.

When not to use

  • Inside small self-contained components (e.g., buttons, chips, individual cards).
  • For icon alignment or micro-layouts that are better handled with intrinsic spacing tokens.
  • For one-off marketing layouts that intentionally break the grid (only with design approval).
  • For fixed-width components that don't need responsive behavior.

Dos & Don'ts

Do

  • Align major content boundaries to grid columns at each breakpoint.
  • Use consistent gutters and margins defined by spacing tokens.
  • Design first in the Figma Grid System file to validate column usage.
  • Respect the container widths and margins specified for each breakpoint.

Don't

  • Hard-code pixel-based widths that ignore the grid.
  • Mix different grid interpretations (e.g., custom columns) on the same page.
  • Place critical content outside the grid or flush to the viewport edge without intent.
  • Override grid margins or gutters with arbitrary spacing values.

Anatomy

The GCC grid system consists of the following core parts:

  • Viewport width – the device or browser width (e.g., 375px, 768px, 1280px).
  • Columns – vertical divisions that content aligns to (count varies by breakpoint: 4 for mobile/tablet, 10 for desktop).
  • Gutters – space between columns (16px for mobile, 24px for tablet/desktop).
  • Margins – space between the outermost columns and the viewport edge (16px for mobile, 32px for tablet, 48px for desktop).
  • Container – the content area width within the grid (343px for mobile, 704px for tablet, 1183px for desktop).

Grid Specifications

Breakpoint Viewport Width Columns Gutter Margin Container Width
Mobile 375px 4 16px 16px 343px
Tablet 768px 4 24px 32px 704px
Desktop 1280px 10 24px 48px 1183px

Behavior & States

The grid system is responsive: it adapts column counts, gutters, and margins as the viewport crosses defined breakpoints. At narrower widths, layouts stack; at wider widths, content fans out across more columns.

  • Mobile (375px) – 4 columns with 16px gutters and margins; content area is 343px wide. Primary content stacks vertically with safe margins.
  • Tablet (768px) – 4 columns with 24px gutters and 32px margins; content area is 704px wide. Two or more columns allow side-by-side modules (e.g., nav + content).
  • Desktop (1280px) – 10 columns with 24px gutters and 48px margins; content area is 1183px wide. Full grid used for navigation, content, and sidebars.

When designing interactions (drawers, modals, slide-overs), anchor them to the grid so that their widths and positions feel aligned with the underlying layout. Modal and drawer widths should respect grid margins and container constraints at each breakpoint.

Variants

The grid system supports a small set of layout variants based on viewport and content type:

  • Contained grid – content constrained to the container width, centered with margins (all breakpoints).
  • Full-bleed sections – background color or imagery extends to the viewport edge, while content still aligns to the grid columns.
  • Single-column focus – mobile layouts that prioritize one column of content with clear margins.
  • Multi-column layouts – desktop layouts that span multiple columns for complex content arrangements.

Choose variants based on the complexity of the content: simple marketing sections can use full-bleed, whereas dense product and account layouts should use contained grids with proper column alignment.

Accessibility

  • Keep line lengths within comfortable ranges (typically ~60–80 characters on desktop).
  • Ensure tap and click targets maintain spacing even on narrow mobile grid configurations (minimum 44×44px touch targets).
  • Avoid horizontally scrolling layouts except for clearly signposted carousels or tables.
  • Respect user zoom and OS font scaling—grid-based layouts must not break when text size increases.
  • Maintain sufficient contrast between content and background, especially when using full-bleed variants.

Tokens & Theming

Grid behavior should be driven by layout tokens and not hard-coded values. Typical tokens include:

  • layout.grid.columns.mobile (4), layout.grid.columns.tablet (4), layout.grid.columns.desktop (10)
  • layout.grid.gutter.mobile (16px), layout.grid.gutter.tablet (24px), layout.grid.gutter.desktop (24px)
  • layout.grid.margin.mobile (16px), layout.grid.margin.tablet (32px), layout.grid.margin.desktop (48px)
  • layout.grid.container.mobile (343px), layout.grid.container.tablet (704px), layout.grid.container.desktop (1183px)

Implement these as CSS variables or theme values so layouts can be adjusted centrally. Do not hard-code pixel widths or ad‑hoc media queries in individual components.

See Spacing for the spacing scale and how gutters/margins relate to global spacing tokens.

Code & API

The grid system can be implemented using CSS Grid aligned to layout tokens. Example:

<!-- Layout shell -->
<main class="page">
  <section class="grid">
    <div class="grid-item">...content...</div>
    <div class="grid-item">...content...</div>
  </section>
</main>

/* Example CSS using tokens */
.grid {
  display: grid;
  grid-template-columns: repeat(var(--layout-grid-columns, 4), minmax(0, 1fr));
  column-gap: var(--layout-grid-gutter);
  margin-inline: var(--layout-grid-margin);
  max-width: var(--layout-grid-container);
  margin-inline: auto;
}

/* Mobile (default) */
:root {
  --layout-grid-columns: 4;
  --layout-grid-gutter: 16px;
  --layout-grid-margin: 16px;
  --layout-grid-container: 343px;
}

/* Tablet */
@media (min-width: 768px) {
  :root {
    --layout-grid-columns: 4;
    --layout-grid-gutter: 24px;
    --layout-grid-margin: 32px;
    --layout-grid-container: 704px;
  }
}

/* Desktop */
@media (min-width: 1280px) {
  :root {
    --layout-grid-columns: 10;
    --layout-grid-gutter: 24px;
    --layout-grid-margin: 48px;
    --layout-grid-container: 1183px;
  }
}

Integrate these tokens into your design system theme so that page templates and high-level layouts can consume them without redefining breakpoints. Always use the container width to constrain content at each breakpoint.

Examples

  • PDP layout: Image gallery, product info, and recommendations aligned to desktop grid columns (10 columns), stacking on mobile (4 columns).
  • Checkout flow: Multi-step forms constrained to the container width (343px mobile, 704px tablet, 1183px desktop) with consistent margins at each breakpoint.
  • Account dashboard: Card-based modules arranged in 2–3 columns on desktop (spanning multiple of the 10 columns) and a single column on mobile.
  • Product listing: Grid of product cards that adapts from 1 column (mobile) to 2–3 columns (tablet) to 4–5 columns (desktop) while respecting grid alignment.

Refer to the Figma Grid System file for canonical examples of how content should align to the grid at each viewport width (Mobile 375px, Tablet 768px, Desktop 1280px).
